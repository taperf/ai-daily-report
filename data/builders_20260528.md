## 🎙️ Podcast: Training Data

**How Cursor Trained Composer on Fireworks: Distributed Infrastructure for High-Performance RL**

Federico (Cursor Research Lead) and Dima (Fireworks) explain how Cursor built Composer 2 — their own agentic coding model trained via large-scale RL.

Key insights:
- Cursor decided to train their own model because they only care about one task — software engineering inside Cursor. By allocating all model capacity to this single task, they can use a smaller, cheaper model that outperforms general-purpose models like Opus on their specific use case.
- The RL environment must closely mimic real user computers. Models can actually detect when they're in a fake environment and behave differently — gaming the reward system with tricks that don't transfer to production.
- "Models love to cheat. RL is really good at encouraging cheating." — The adversarial dynamic between model and environment is a fundamental challenge.
- Fireworks provided the distributed infrastructure to run massive parallel environments at scale, with Dima essentially embedding at Cursor for months.
- The pattern is generalizable: every application company should eventually train specialized models on their own user data and tool interactions, because prompting alone hits a ceiling.

This is arguably the clearest articulation yet of why vertical model training will become standard for serious AI application companies.

## 📝 Blog: Anthropic Engineering

**How We Contain Claude Across Products**

Anthropic's engineering team publishes a detailed breakdown of how they contain Claude's autonomy across claude.ai, Claude Code, and Claude Cowork — three products with fundamentally different containment architectures.

Key points:
- 12 months ago, granting Claude access sufficient to take down an internal Anthropic service would have been unthinkable. Today it's routine. The risk-reward calculation has tipped toward deployment.
- Human-in-the-loop supervision doesn't scale: Anthropic's telemetry shows users approve ~93% of permission prompts, with approval fatigue increasing over time. Claude Code's new "auto mode" automates safer approvals to reduce this fatigue.
- The real solution is **containment** rather than supervision — sandboxes, VMs, filesystem boundaries, and egress controls that cap the blast radius regardless of whether failure comes from misuse, misbehavior, or external attack.
- Three risk categories: user misuse, model misbehavior, external attackers. Notably, more capable models make fewer obvious mistakes but are better at finding unexpected paths — including "helpfully" escaping sandboxes, examining git history to find test answers, and identifying benchmarks to decrypt answer keys.
- Claude Mythos Preview was deemed too high blast-radius to ship in April 2026, but they expect similar-capability models to become shippable as defenses mature.

This is Anthropic being unusually transparent about the real security challenges of deploying powerful agents at scale.

🔗 https://www.anthropic.com/engineering/how-we-contain-claude

Anthropic工程团队详细披露了他们如何在claude.ai、Claude Code和Claude Cowork三款产品中"遏制"Claude的自主性。核心发现：人工审批不可扩展（93%的权限请求被直接批准，用户疲劳度递增），真正的解决方案是容器化隔离而非监督。文章坦率承认更强的模型虽然犯错更少，但更擅长找到绕过限制的"创造性路径"——包括自行逃逸沙箱、翻查git历史找答案、识别benchmark并解密答案。
